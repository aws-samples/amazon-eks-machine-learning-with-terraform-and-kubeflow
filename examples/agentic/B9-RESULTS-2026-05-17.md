## D5 — cross-agent provenance chain (verified)

```
chain_depth     = 2
min_confidence  = 0.85
agents_involved = ['eks-ops-agent', 'triage-agent']
hops:
  hop 0 origin   eks-ops-agent (confidence 0.85) — remediation
  hop 1 derived  triage-agent  (confidence 0.90) — root incident
```

Weakest-link semantics confirmed: even though the head was 0.85, the
chain min is 0.85 because the only ancestor was 0.90 — meaning if any
ancestor had been lower, it would have pulled the effective confidence
down.

## D6/D8 — 3-tier MAI evaluator suite

| Scenario | Ground truth | Option A (deterministic) | Option B (structural) | Option C (RAGAS LLM-judge) |
|----------|--------------|--------------------------|------------------------|----------------------------|
| w01 (well-attributed) | well_attributed | 1.000 PASS ✓ | 1.000 PASS ✓ | 1.000 PASS ✓ |
| w02 (poorly-attributed, conf=0.3, hedged, no chain) | poorly_attributed | 0.875 PASS ✗ | 0.800 PASS ✗ | **0.000 FAIL ✓** |
| w03 (well-attributed, multi-agent chain) | well_attributed | 1.000 PASS ✓ | 1.000 PASS ✓ | 1.000 PASS ✓ |

Tier-2 RAGAS LLM-judge correctly catches the w02 poorly-attributed case
that tiers A and B let through at the default threshold of 0.7. This is
what the 3-tier design is for: deterministic rules and structural checks
are cheap and agree with the LLM in the obvious cases, and the LLM-judge
acts as the tiebreaker when the rule-based evaluators are too lenient.
Judge model used: `bedrock/us.anthropic.claude-sonnet-4-20250514-v1:0`
via LiteLLM. Eval extras (`memledger[eval]`) are not in the agent images
by default — installed ad-hoc for this run; consider adding to the
Dockerfile if continuous evaluation in the cluster is desired.

Recommendation for v1.1: tighten the default deterministic + structural
thresholds (e.g. raise to 0.85 or weight `confidence` more) so the
cheaper tiers also catch w02-style cases.

## D7 — `record_outcome()`

`record_outcome(record_id, success=...)` increments `success_count` /
`failure_count` columns on the record. Confirmed: 2 success calls + 1
failure call → `success_count=2 failure_count=1`. The CHANGELOG describes
this as feeding into a confidence-update loop; v1.0.0 collects the raw
counts and the auto-promote logic uses them at retrieval time
(`auto_promote_threshold=3` by default), but `record.confidence` is not
mutated in-place by `record_outcome()` itself.

## D10 — Confidence FLAG signal

`ConfidencePolicy(min_threshold, flag_threshold)` exposes per-search
aggregate counts via `SearchResult.metadata.confidence_gating`:

```python
metadata = {
    "confidence_gating": {
        "passed": 4,        # confidence >= flag_threshold
        "flagged": 4,       # min_threshold <= confidence < flag_threshold
        "filtered": 4,      # confidence < min_threshold (excluded)
        "policy": {"min": 0.4, "flag": 0.7},
        "uses_effective_confidence": True,  # B10 weakest-link
    }
}
```

The aggregate model (counts, not per-record decoration) is fine for
audit dashboards but means callers cannot distinguish a PASS record
from a FLAG record in a single search response — they'd have to compare
each `record.confidence` against `flag_threshold` themselves. Worth
exposing per-record signals in v1.1 for callers that want to surface
hedged warnings inline.

## D11 — RBAC enforcement through `Memledger.search(requester_id=...)`

| Test case | Expected | Actual |
|-----------|----------|--------|
| `triage-agent` → `/triage/incidents/payment-oom` (rule allows read) | ALLOW | 5 records returned |
| `eks-ops-agent` → `/triage/incidents/payment-oom` (no rule, default deny) | DENY | 0 records (silently filtered) |
| `compliance-agent` → `/triage/incidents/payment-oom` (rule allows read) | ALLOW | 5 records returned |
| `eks-ops-agent` → `/eks-ops/remediations/payment` (no policy attached → default allow) | ALLOW | 5 records returned |

RBAC enforcement is **silent** — denied calls return 0 records and
`metadata.rbac` is empty. Caller cannot distinguish "no matches found"
from "denied by policy". For v1.1 consider:

- populating `SearchResult.metadata.rbac` with `{denied: True, reason: ...}`
  when the requester is denied;
- raising `PermissionError` for explicit-deny rules so callers can
  branch on it without having to inspect record counts.

The `Memledger.rbac` property is read-only after construction; the only
documented path to attach a policy at runtime is via the `rbac=` kwarg
on `Memledger.__init__()`. We bypassed that in this test by setting
`ml._rbac = ...` directly. Consider adding a setter or
`Memledger.set_rbac(rbac)` for runtime policy updates.

## Code fixes required to get to PASS

These all live in the agent integration layer in this repo, not in the
memledger SDK. v1.0.0 SDK behaved correctly throughout — the prior
0.4.0-era integration code did not surface the v1 attribution surface.

1. `Dockerfile` — drop `aws-opentelemetry-distro` (AWS-specific, not OSS
   v1); drop `aws-otel-distro` env vars and `opentelemetry-instrument`
   wrapper.
2. `pyproject.toml` — pin `a2a-sdk<1.0` because `kagent-langgraph==0.7.19`
   declares `a2a-sdk>=0.3.23` with no upper bound and `a2a-sdk 1.0`
   reorganized the `a2a.server.apps` namespace.
3. `build-and-deploy.sh` — let an externally set `MEMLEDGER_PG_DSN` win
   over the auto-resolved one from `kagent-db-credentials`.
4. `src/memory.py` `remember_knowledge` tool — accept and pass through v1
   trust kwargs: `confidence`, `hedged`, `derived_from`, `supersedes`,
   `agent_id`, `created_by`, `workflow_id`, `triggered_by`.
5. `src/memory.py` `recall_knowledge` tool — print full record UUIDs so
   downstream agents can pass them to `derived_from` (the prior
   8-char-prefix display caused agents to copy truncated IDs).
6. `src/app.py` — initialize an explicit `TracerProvider` +
   `BatchSpanProcessor(OTLPSpanExporter)` at startup. Without this the
   default `ProxyTracerProvider` is a no-op, so memledger spans were
   never exported.
7. `src/memory.py` `MemoryService._get_*()` — call
   `instrument_engram(self._ml)` after creating the Memledger instance.
   v1's tracing is **opt-in** at the instance level by design.
